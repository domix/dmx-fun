The `fun-tracing` module instruments `Try` and `Result` executions with distributed tracing
spans automatically, opening a named child span for each call and recording outcome signals
without requiring manual `Span.start()` / `Span.end()` management at call sites. It is an
**optional** dependency — the core `fun` library has no runtime dependency on Micrometer Tracing.

`fun-tracing` declares `micrometer-tracing` as `compileOnly`. You bring your own version
and bridge (OpenTelemetry or Brave), following the same pattern as `fun-micrometer` and
`fun-resilience4j`. Any version from **1.2.x through 1.6.x** is supported
(see [version compatibility](#version-compatibility) below).

## Signals recorded

For every instrumented call, the following signals are written to the active span:

| Signal | When |
|---|---|
| Span named `{name}` | Always — the span name is the `name` argument |
| `outcome=success` tag | Supplier returned normally |
| `outcome=failure` tag | Supplier threw any exception |
| `exception=<SimpleClassName>` tag | Supplier threw — simple class name of the cause |
| Span marked as error | Supplier threw — recorded via `span.error(cause)` |

The span is always ended before the method returns, even when the supplier throws.

## Span naming convention

Use stable, low-cardinality operation names. A practical convention is:

`<bounded-context>.<operation>`

Examples:
- `payment.charge`
- `inventory.reserve`
- `order.place`
- `http.client.get`

Avoid embedding IDs or user input in span names — those belong in log fields or baggage,
not in span names, as high-cardinality names generate excessive trace index entries in
Zipkin, Jaeger, and similar backends.

## `exception` tag cardinality

The `exception` tag uses the simple class name of the thrown exception (e.g. `IOException`,
`TimeoutException`). Keep this cardinality bounded:

- Prefer stable exception classes from your own domain or the JDK.
- Avoid dynamically generated exception types.
- Do not encode request-specific values in exception class names.

## `DmxTracing`

The primary entry point. Create one instance per `Tracer` (typically application-scoped)
and call `traceTry` or `traceResult` at each instrumentation point.

## `DmxTraced` — fluent builder

## Choosing `DmxTracing` vs `DmxTraced`

| Scenario | Recommendation |
|---|---|
| Multiple operations in the same class | `DmxTracing.of(tracer)` — inject once, call many times |
| Single ad-hoc operation or factory method | `DmxTraced.of(name).tracer(tracer).traceTry(...)` |
| Spring Boot bean | Inject `Tracer`, wrap with `DmxTracing.of(tracer)` in the constructor |

## Real-world example

An order service where every step — inventory reservation, payment charge, and
persistence — is instrumented as an independent child span. No manual span bookkeeping
in the business logic; every step appears in the trace timeline automatically.

## Spring Boot integration

Spring Boot auto-configures a `Tracer` bean when a Micrometer Tracing bridge is on the
classpath. Inject it directly:

```java
@Service
public class PaymentService {

    private final DmxTracing dmx;

    public PaymentService(Tracer tracer) {
        this.dmx = DmxTracing.of(tracer);
    }

    public Try<Receipt> charge(Amount amount, PaymentToken token) {
        return dmx.traceTry("payment.charge",
            () -> gateway.charge(amount, token)
        );
    }
}
```

No additional configuration is required — the bridge wires `Tracer` to the active
propagation context (W3C TraceContext, B3, etc.) automatically.

## Version compatibility

The module is tested in CI against the following Micrometer Tracing versions on every
pull request that touches the `tracing/` module:

| Micrometer Tracing | Spring Boot | Status  |
|--------------------|-------------|---------|
| 1.6.x              | 3.5.x       | tested  |
| 1.5.x              | 3.4.x       | tested  |
| 1.4.x              | 3.3.x       | tested  |
| 1.3.x              | 3.2.x       | tested  |
| 1.2.x              | 3.1.x       | tested  |

To test locally against a specific version:

```bash
./gradlew :tracing:test -PmicrometerTracingVersion=1.5.11
```
