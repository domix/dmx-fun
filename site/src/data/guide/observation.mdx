The `fun-observation` module instruments `Try` and `Result` executions using the
**Micrometer Observation API** (`micrometer-core` ‚â• 1.10). A single
`observeTry` or `observeResult` call produces **both** metrics and distributed
tracing spans, dispatched through whichever `ObservationHandler`s are registered
‚Äî no manual `counter.increment()`, `timer.record()`, or `Span.start()` calls
required in your business logic.

The module is **optional** ‚Äî the core `fun` library has no runtime dependency on
Micrometer. `fun-observation` declares `micrometer-core` as `compileOnly`; you
bring your own version and backend. Any version from **1.10.x through 1.16.x**
is supported (see [version compatibility](#version-compatibility) below).

> **`fun-observation` vs `fun-tracing`** ‚Äî Both modules instrument `Try`/`Result`
> operations, but they target different Micrometer layers:
>
> | Module | API used | What you get |
> |---|---|---|
> | `fun-observation` | Micrometer Observation API | Metrics **and** spans from one call ‚Äî the recommended choice |
> | `fun-tracing` | Micrometer Tracing API | Spans only ‚Äî use when you already depend on `micrometer-tracing` directly |
>
> Prefer `fun-observation` for new projects. It sits at a higher abstraction layer
> and will automatically emit metrics, spans, or both depending on the handlers in
> your `ObservationRegistry` ‚Äî with no code changes required when you add or change
> backends.

## Signals recorded

For every instrumented call, the following signals are dispatched through the
registered `ObservationHandler`s:

| Signal | When |
|---|---|
| Timer `{name}_seconds` | Always ‚Äî tagged `outcome=success\|failure` |
| Span named `{name}` | When a tracing bridge (`micrometer-tracing-bridge-otel` or `-brave`) is configured |
| `outcome=success\|failure` | Always ‚Äî low-cardinality key on both metric and span |
| `exception=<label>` | On failure ‚Äî low-cardinality key on both metric and span |
| Observation marked as error | On failure ‚Äî recorded via `observation.error(cause)` |

The observation is always stopped before the method returns, even when the
supplier throws.

## Exception key cardinality

> **Warning:** The default `exception` key uses `getClass().getSimpleName()`. When
> arbitrary third-party exceptions can appear at runtime, this produces an unbounded
> number of distinct values ‚Äî a violation of Micrometer's `lowCardinalityKeyValue`
> contract (‚â§ 100 distinct values per key). This causes excessive series growth in
> Prometheus, Zipkin, and similar backends.

**In production, always supply an explicit classifier** that maps every reachable exception
type to one of a small, fixed set of labels:

```java
DmxObservation dmx = DmxObservation.of(observationRegistry, cause ->
    switch (cause) {
        case IOException _           -> "io";
        case TimeoutException _      -> "timeout";
        case IllegalStateException _ -> "state";
        default                      -> "other";
    }
);
```

The same option is available on the `DmxObserved` builder:

```java
DmxObserved.of("payment.charge")
    .registry(observationRegistry)
    .exceptionClassifier(cause -> cause instanceof IOException ? "io" : "other")
    .observeTry(() -> stripe.charge(amount));
```

## Observation naming convention

Use stable, low-cardinality operation names. A practical convention is:

`<bounded-context>.<operation>`

Examples:
- `payment.charge`
- `inventory.reserve`
- `order.place`
- `db.user.find`

Avoid embedding IDs or user input in observation names ‚Äî those belong in log
fields or baggage, not in metric/span names, as high-cardinality names generate
excessive entries in Prometheus, Zipkin, Jaeger, and similar backends.

## `DmxObservation`

The primary entry point. Create one instance per `ObservationRegistry` (typically
application-scoped) and call `observeTry` or `observeResult` at each
instrumentation point.

## Choosing `DmxObservation` vs `DmxObserved`

| Scenario | Recommendation |
|---|---|
| Spring Boot with `fun-spring-boot` | Inject `DmxObservation` directly ‚Äî auto-configured, no setup needed |
| Multiple operations in the same class (plain Spring) | `DmxObservation.of(registry)` ‚Äî inject `ObservationRegistry`, wrap once in the constructor |
| Single ad-hoc operation or factory method | `DmxObserved.of(name).registry(registry).observeTry(...)` |

classpath, a child span. No manual instrumentation in the business logic.

## Spring Boot integration

When `fun-spring-boot` and `fun-observation` are both on the classpath, Spring
Boot registers a `DmxObservation` bean automatically ‚Äî no `@Bean` declaration or
manual wiring required. Inject it directly into your services:

<SpringBootAuto/>

### How it works

`DmxFunObservationAutoConfiguration` activates when:

1. `micrometer-core` and `fun-observation` are on the classpath (`@ConditionalOnClass`).
2. An `ObservationRegistry` bean is present in the context (`@ConditionalOnBean`) ‚Äî
   Spring Boot registers one automatically via `ObservationAutoConfiguration`.

The `DmxObservation` bean is backed by that `ObservationRegistry`, so all signals
it records inherit the active observation context without any extra wiring.

### Opting out

```properties
dmx.fun.observation.enabled=false   # disable DmxObservation auto-configuration
```

### Replacing the bean

The bean is guarded by `@ConditionalOnMissingBean`. Declare your own `@Bean` of
type `DmxObservation` and the auto-configuration backs off:

```java
@Bean
DmxObservation dmxObservation(ObservationRegistry registry) {
    // Supply an explicit classifier in production ‚Äî see "Exception key cardinality" above.
    return DmxObservation.of(registry);
}
```

### Without `fun-spring-boot`

If you use plain Spring (no `fun-spring-boot`), construct `DmxObservation` manually
in your `@Bean` or constructor:

```java
@Service
public class PaymentService {

    private final DmxObservation dmx;

    public PaymentService(ObservationRegistry registry) {
        // Supply an explicit classifier in production ‚Äî see "Exception key cardinality" above.
        this.dmx = DmxObservation.of(registry);
    }
}
```
